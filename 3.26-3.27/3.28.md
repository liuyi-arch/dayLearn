# useCallback

### 使用场景：
- 该函数作为props，传递给已被React.memo包裹的子组件；
- 该函数作为其他hook的依赖；
- 防抖/节流函数中（确保重渲染不会创建新的计时器）；
> Q1：子组件没被React.memo包裹行吗？为什么？
> 
> - 子组件没有使用React.memo，那么即使父组件使用useCallback保证函数引用不变，子组件也会因为父组件重新渲染而渲染；
> 
> Q2：React.memo与useMemo区别？
>
>  - React.memo：它是一个高阶组件，包裹整个组件，通过对比props是否改变来判断是否跳过父组件渲染而引起的子组件渲染；
>  - useMemo：是一个hook，它用于包裹一段逻辑或值，通过对比前后依赖项来判断是否重新计算；

### 从原理角度，说明useCallback是如何缓存函数引用，避免不必要的重复渲染的：
- 首次渲染时，useCallback会将函数和依赖项存储在React Fiber节点的memoizedState中；
- 后续渲染，通过浅比较前后依赖项，来决定是否创建新的函数实例；
> Q1：什么是浅比较？
>
> - 对于原始类型，对比值是否相等；
> - 对于引用类型，只对比引用是否相等，而不递归遍历内部属性值是否全部相等；

### 结合实例说明，如何确定useCallback中依赖数组内容：
- 所有在函数体内使用且来自组件作用域的变量都必须放入依赖数组；
- 如果不放，函数内部会形成闭包陷阱，引用的永远是函数创建时的旧值；

# useMemo

### 使用场景：
- 复杂计算中缓存计算结果，比如列表过滤、排序等；
- 该对象作为子组件的props和其他hook的依赖项时；

### 从原理角度，说明useMemo是如何缓存计算结果，避免重复计算执行的：
- 首次渲染时，useMemo将计算结果和依赖项存储在React Fiber节点的memoizedState中；
- 后续渲染，通过比较前后依赖项，来决定是否重新计算结果；

### 如何确定useMemo中依赖数组内容：
- 该计算函数所有input；

### useMemo依赖项是一个对象或数组，如何处理：
- 将该对象也用useMemo包裹；
- 将对象拆解，依赖项由[obj] -> [obj.key1, obj.key2, ...]；
> Q1：如何理解尽量避免在依赖项中使用深度比较？为什么？
>
> - 深比较一个大对象，可能比重新计算结果开销更大，所以react只内置了浅比较；
> - 可以通过上述两种方式处理依赖项是obj情况，即使用useMemo包裹obj、将obj属性平铺；

# useCallback与useMemo对比

### 返回值及用途：
- useCallback返回函数，useMemo返回计算结果；
- useCallback缓存函数引用，useMemo缓存计算结果；
- 本质联系：useCallback(fn, deps) 实际上等价于 useMemo(() => fn, deps)；
> Q1：如何理解useCallback(fn, deps) 实际上等价于 useMemo(() => fn, deps)？
>
> - useCallback缓存的是第一个参数，useCallback(fn, deps)中，即将fn存入Fiber节点的memoizedState；
> - useMemo缓存的是第一个参数的计算结果，useMemo(() => fn, deps)中，() => fn执行结果即fn；
>
> Q2：使用useCallback、useMemo分别实现对方？
>
> ```js
> function myUseCallback(fn, deps){
>   return useMemo(() => fn, deps);
> }
>
> function myUseMemo(fn, deps){
>   const temp = useCallback(fn, deps);
>
>   return temp();
> }
> ```

### useCallback和useMemo是否可能导致内存泄漏，原因是什么：
- 被缓存的函数引用组件作用域内的变量，如果依赖项长期不改变，那么这些资源将无法被垃圾回收；
- 但是组件卸载时，资源会被释放；

### 如何在多个组件中共享useCallback或useMemo的逻辑：
- 将useCallback或useMemo封装在自定义hook中，作为该hook返回值返回；
- 通过context向下传递useCallback或useMemo的返回值；
